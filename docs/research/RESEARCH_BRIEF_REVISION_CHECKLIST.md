# Research Brief Revision Checklist

Status: Working checklist
Last updated: 2026-04-21
Purpose: Update the research brief after interviews without rewriting everything from scratch.

## When to Use

Use this checklist after you complete:

- 2 student interviews
- 1 coach interview
- 1 page synthesis using [INTERVIEW_SYNTHESIS_TEMPLATE.md](INTERVIEW_SYNTHESIS_TEMPLATE.md)

## Step 1: Read Before Editing

Open these files first:

- [RESEARCH_BRIEF.md](RESEARCH_BRIEF.md)
- [INTERVIEW_SYNTHESIS_TEMPLATE.md](INTERVIEW_SYNTHESIS_TEMPLATE.md)
- [RELATED_WORK.md](RELATED_WORK.md)

Do not start editing from memory only.

## Step 2: Update the Highest-Impact Sections First

Revise these sections in this order:

1. `Problem Statement`
2. `Who & Why Now`
3. `Evidence Assessment`
4. `Killer Scenarios`
5. `Expected Contribution`
6. `Anticipated Reviewer Objections`

Do not start by editing titles or wording polish.

## Step 3: Replace Generic Claims with Evidence

For each of these claims, ask:

- Do I now have a quote or episode that supports this?
- If not, should I weaken or remove it?

Common replacements:

- replace `students lack feedback` with a concrete workaround failure
- replace `students want AI coaching` with an adoption condition from interviews
- replace `this would help coaches` with a coach quote or narrow the claim

## Step 4: Check the Primary User

After the interviews, confirm:

- Is the primary user still the solo-practice student?
- Or did evidence shift toward coaches, team practice, or post-practice review?

If the answer changed, revise all sections consistently.

## Step 5: Check the System Role

Make sure the brief consistently treats the system as one of these:

- solo-practice aid
- coach assistant
- review tool

Do not mix all three unless interviews really support that breadth.

## Step 6: Tighten the Problem Statement

A good revised problem statement should include:

- one concrete practice context
- one common workaround
- one concrete failure
- one concrete cost

If the problem statement still sounds generic, it is not revised enough.

## Step 7: Re-evaluate the Main Claim

Ask:

- Is this still an `experience gap`?
- Or did interviews reveal it is actually a `cost gap` or `workflow gap`?

Keep the claim that best matches the new evidence, not the original wording.

## Step 8: Revise the Killer Scenarios

Update the scenarios so they match what participants actually said.

Good revision signs:

- scenario uses a real workaround from interviews
- scenario addresses a real missed mistake
- scenario reflects a real adoption condition

Bad revision signs:

- scenario is still just a system demo
- scenario includes features nobody asked for

## Step 9: Update Reviewer Objections

After interviews, revise objections to match the real remaining weaknesses.

Examples:

- if users liked the idea but worried about friction, add an adoption objection
- if coaches doubted credibility, add a trust objection
- if students mostly wanted post-hoc replay, weaken the real-time claim

## Step 10: Final Consistency Check

Before you stop, check:

- Does the intro match the evidence?
- Does the primary user stay the same throughout?
- Does the claimed contribution match the system role?
- Are there any claims that still sound stronger than the data?

## Quick Revision Questions

Answer these after editing:

- What is now the strongest evidence in the brief?
- What claim became weaker after interviews?
- What sentence should be removed because it is no longer supported?
- What sentence became much better because of a direct quote?

## Done Condition

The brief is ready for the next round if:

- the main problem is supported by at least one concrete episode
- the main workaround failure is clearly described
- the primary user is unambiguous
- the main claim is not stronger than the evidence
